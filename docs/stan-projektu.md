# Stan projektu

**Żywy dokument.** Każdy agent po zakończeniu zadania aktualizuje ten plik tak, żeby odzwierciedlał rzeczywisty, bieżący stan systemu — nie dopisuje kolejnych wpisów dziennika, tylko poprawia treść. Historia jest w gicie.

Ostatnia aktualizacja: 2026-07-14, po zakończeniu piętnastu audytów fundamentu. Produkt jest platformą Sklepik do uruchamiania niezależnych sklepów; sklep kakao pozostaje pierwszym tenantem referencyjnym. Edytor strony głównej z bezpiecznym draft/publish, publiczny renderer w `sklepikFront`, panel dokumentów prawnych oraz status `draft/live` z checklistą gotowości i backendową blokadą przyjęcia pieniędzy są zmergowane i wdrożone: backend na Oracle, panel i storefront na Vercelu. Historyczne buildy, Brakeman, SDK, Dashboard E2E oraz smoke signup/storefrontu przeszły, ale audyty wykazały krytyczne luki, których te testy nie obejmowały. **System nie jest gotowy do niezależnych merchantów, prawdziwych danych klientów, płatności online ani samodzielnego fulfillmentu.** Publiczny signup pozostaje funkcją eksperymentalną i nie powinien być szeroko otwierany. Zewnętrzne blokery — operator płatności, prawdziwe dane i treści prawne merchanta oraz produkcyjny test zamówienia — nadal obowiązują, lecz nie są już jedynymi blokerami.

**Dopisek 2026-07-17 — kierunek "repo/deployment per sklep" opisany wyżej i w całym tym dokumencie jest PORZUCONY** (decyzja właściciela po dziewięciu research-passach nad wzorcami multi-tenant SaaS). Nowy model docelowy: jeden współdzielony storefront, layout jako dane — `docs/plans/storefront-composition-system.md`. Wszystko, co ten dokument opisuje jako "Store Factory"/signup/provisioning/edytor w `packages/dashboard`, jest legacy: zweryfikowany, częściowo wdrożony, ale nierozwijany dalej. Szczegóły i uzasadnienie: `docs/plans/store-factory.md` (nota u góry) i sekcja "Co działa" niżej.

## Stan zweryfikowany programem audytu 2026-07-14

Program objął wszystkie **4 859/4 859 śledzonych ścieżek** w obu repozytoriach na baseline'ach backend/admin `9a4f693147` i storefront `0f83b94`, a następnie piętnaście raportów obszarowych: inwentaryzację, architekturę, granicę silnika, tenant isolation, auth, bezpieczeństwo, pieniądze, cykl zamówienia, bazę, DR, async, infrastrukturę, API/SDK, panel/onboarding i storefront. Raport nadrzędny i źródło zdeduplikowanej oceny to [`docs/audits/2026-07-14-00-stan-fundamentu-sklepika.md`](audits/2026-07-14-00-stan-fundamentu-sklepika.md); raporty 01–15 są dowodem szczegółowym, nie piętnastoma niezależnymi backlogami.

Zweryfikowane mocne strony:

- commerce pozostaje po stronie backendu; storefront nie jest źródłem cen, stocku ani reguł zamówienia;
- Store API wybiera tenant publishable keyem i kontroluje konflikt host/store, Admin API wymaga membershipu, a cache uwzględnia kontekst sklepu;
- draft layoutu nie wycieka publicznie, publiczny renderer czyta wyłącznie opublikowany snapshot, a nowy sklep ma backendowy gate `draft/live`;
- typowe zasoby katalogu, koszyk, zamówienie i większość konfiguracji mają scope sklepu, blokady rekordów i sensowne podstawy testowe;
- istnieją osobny Sidekiq, rekordy dostaw webhooków, HMAC, backup dzienny Postgresa i podstawowe hardening hosta.

Zweryfikowane ograniczenia tej oceny:

- większość raportów była statyczna; nie było destrukcyjnego testu produkcji, prawdziwego PSP, restore'u, chaos testu ani pełnego black-box E2E dwóch tenantów;
- część testów nie wystartowała w środowisku audytu z powodu braku `pnpm` albo zapisywalnej testowej bazy; taki wynik oznacza **nieweryfikowane**, nie zielone;
- „kod istnieje”, „wdrożone” i „zweryfikowane runtime” są od teraz trzema osobnymi statusami.

### Krytyczne blokery wykryte po audytach

1. **Izolacja i przejęcie konta (P0):** `Spree.user_class.for_store` jest globalne, a standardowy admin ma `manage :all`; admin sklepu A może zobaczyć lub zmienić klientów i PII sklepu B (`TENANT-001`). Produkcyjne konto administracyjne używało jawnego, słabego credentialu opisanego w dokumentacji i wymaga natychmiastowej rotacji oraz unieważnienia sesji (`AUTH-001`).
2. **Pieniądze i webhook PSP (P0):** klient publicznego API kontroluje kwotę `PaymentSession` bez związania z `order.amount_due` (`MONEY-001`); generowany URL webhooka wskazuje storefront zamiast backendu (`MONEY-002`); wspólny host webhooka nie rozpoznaje pewnie metody drugiego tenanta (`MONEY-003`); endpoint może odpowiedzieć `200` przed trwałym inboxem lub mimo błędu (`ASYNC-001`/`MONEY-007`).
3. **Release, migracje i artefakt produkcyjny (P0):** migracje efemerycznego host-app nie są trwałym, jednoznacznym artefaktem (`DB-001`); deploy nie czeka na testy i ignoruje błąd budowy obrazu (`INFRA-001`); produkcja buduje inny, niepowtarzalny obraz z nieprzypiętego upstreamu (`INFRA-002`).
4. **Blokery P1 przed prawdziwą sprzedażą:** per-store customer identity i reset, revocation/digest sesji, owner verification/recovery; sanitizacja rich text i zamknięcie SSRF; atomowa idempotencja i trwały outbox/inbox; realny adapter PSP i reconciliation; spójne anulowanie/refund/zwrot/reklamacja; poprawny shipping/tax onboarding; restore drill i działające alerty; aktualny owner/storefront/checkout E2E.

Do czasu zamknięcia P0 obowiązuje operacyjna zasada: nie wpuszczać niezależnych merchantów do wspólnego panelu z realnymi klientami, nie włączać płatności online ani automatycznego fulfillmentu, nie promować zmian na produkcję poza kontrolowanym hotfixem i nie traktować checklisty `live` jako certyfikatu gotowości płatniczej.

### Granica Spree po audycie

Spree jest nadal silnikiem, publicznym protokołem i słownikiem UI, nie wymiennym adapterem: storefront importuje `@spree/sdk` w 82 plikach źródłowych, publiczne nagłówki/cookies/env/webhooki noszą nazwę silnika, a klasy `Spree::*` przeciekają do checkoutu i permissions. Naprawa **nie polega na globalnym rename**. Namespace Ruby, 131 tabel `spree_*`, historyczne migracje i działający core pozostają wewnętrzną implementacją. Publiczna migracja idzie od zewnątrz do środka: własne `@sklepik/contracts`, `@sklepik/store-client` i `@sklepik/admin-client`, adapter `adapters/spree`, zakaz nowych przecieków w CI oraz kompatybilne aliasy dla nagłówków, cookies, env i webhooków. Stare nazwy wolno usuwać dopiero po telemetrii, testach migracji sesji/koszyków i pełnym cyklu aktualizacji floty; szczegóły są w F29 roadmapy i audycie 03.

## Co działa

- **Backend na Oracle Cloud** (141.253.103.172) według ostatniej weryfikacji działa: Docker Compose (Rails/Puma + Postgres + Redis + Sidekiq + Nginx), gemy z tego forka (`SPREE_PATH`), migracje silnika zainstalowane i wykonane, baza zseedowana, TLS przez Let's Encrypt na `nip.io`. Własna stabilna domena API nadal nie istnieje. Audyt 12 wykazał, że obecny build/deploy nie jest deterministyczny ani bezpiecznie promowany; „działa” opisuje runtime historyczny, nie gotowość pipeline'u.
- **Panel admina** (`sklepik-gamma.vercel.app`) działa end-to-end: logowanie JWT przez single-origin proxy, zarządzanie produktami. Wpisy cen kanoniczny format `"1234.56"` niezależnie od locale. Upload logo sklepu (Ustawienia → Sklep) działa (F10).
- **Storefront** (`sklepikkk.vercel.app`) działa: rebranding "Kakałowy Sklepik", polski domyślny locale bez prefiksu URL, katalog i strony produktów renderują się z danych Store API. Logo sklepu (gdy wgrane) renderuje się w nagłówku i JSON-LD z publicznego Store API zamiast statycznego env (F10). Wymaga `SPREE_API_URL` + `SPREE_PUBLISHABLE_KEY` ustawionych jako zmienne środowiskowe Vercel produkcyjnie — bez nich `isSpreeConfigured()` po cichu zwraca puste odpowiedzi (żadnego błędu, katalog wygląda po prostu na pusty; to właśnie się stało i kosztowało długie dochodzenie, zanim się okazało że przyczyną nie był cache).
- **Dane:** jeden rynek Polska/PLN/pl (7 demo-rynków usuniętych), 6 produktów kakao zseedowanych przez Admin API, ceny w PLN, media na R2.
- **Główne zdjęcie produktu (galeria/zoom) już się nie przycina do kwadratu (2026-07-12, ⚠️ EKSPERYMENTALNE — wciąż testowane, patrz uwaga niżej):** `Spree::Asset` (`spree/core/app/models/spree/asset.rb`, `UNCROPPED_VARIANT_SIZES`) generuje warianty `large`/`xlarge` przez `resize_to_limit` (zachowuje proporcje) zamiast `resize_to_fill` (crop) — zmiana retroaktywna, działa dla istniejących i nowych produktów bez re-uploadu, bo zmiana parametrów zmienia digest wariantu i wymusza regenerację przy pierwszym żądaniu. Miniatury na liście (`mini`/`small`/`medium`) i `og_image` zostają kwadratowe/przycinane celowo (spójność siatki, wymagania social cards). Frontend (`sklepikFront/src/components/products/MediaGallery.tsx`) zmieniony z `object-cover` na `object-contain` na głównym zdjęciu, żeby CSS nie przycinał ponownie tego, co backend już zachował.
  **⚠️ Uwaga — to odejście od domyślnego zachowania Spree (upstream zawsze crop-to-fill, kwadrat, na wszystkich rozmiarach):** decyzja podjęta na życzenie właściciela po krótkiej dyskusji o standardach e-commerce, nie zweryfikowana jeszcze w praktyce na różnorodnych zdjęciach produktów (np. bardzo wąskie/panoramiczne kadry mogą wyglądać na małym letterboxowanym kwadracie gorzej niż przycięte). **Do zrobienia:** przetestować na kilku realnych zdjęciach o różnych proporcjach (pionowe, panoramiczne, prawie-kwadratowe) i potwierdzić, że letterbox (szare tło) na głównym zdjęciu faktycznie wygląda lepiej niż crop, zanim uzna się to za ostateczne rozwiązanie. Jeśli nie — łatwo cofnąć: przywrócić `resize_to_fill` w `UNCROPPED_VARIANT_SIZES`-owej gałęzi i `object-cover` w `MediaGallery.tsx`.
- **Walidacja produktu:** endpoint `/api/v3/admin/products/:id/readiness` sprawdza status, publikację, ceny, stock, tłumaczenia; strona edycji produktu w dashboardzie pobiera go (`useProductReadiness`) i pokazuje banner ostrzegawczy z listą niespełnionych warunków, gdy `ready: false` — żadna z tych kontroli nie blokuje zapisu, to czysto informacyjne ostrzeżenie.
- Ostatni audyt storefrontu potwierdził lint, TypeScript, parity locale i **97/97 testów jednostkowych**. Build nie został potwierdzony (utknął na kompilacji i został przerwany), a Playwright ma nieaktualny routing `/us/en`; nie ma bieżącego E2E checkoutu.
- **Wielosklepowość — panel admina, Faza 1 (2026-07-12, F25):** Admin API rozwiązuje `current_store` z nagłówka `X-Spree-Store-Id` zamiast zawsze zwracać jeden `default` store; `GET`/`POST /api/v3/admin/stores` listuje sklepy usera i tworzy nowe (twórca automatycznie dostaje rolę admina nowego sklepu). Panel: `StoreSwitcher` pokazuje realną listę sklepów usera, nowa strona `/$storeId/new-store` do zakładania kolejnego sklepu. Zero migracji bazy. Szczegóły i status weryfikacji: `docs/roadmap.md` F25, `docs/plans/multi-store-support.md`.
- **Store Factory (model "repo per sklep") był zbudowany dalej niż wcześniej udokumentowano, ale jest PORZUCONY jako kierunek 2026-07-17** — patrz `docs/plans/storefront-composition-system.md` dla aktywnego modelu docelowego. Do porzucenia: publiczny signup tworzący właściciela/tenant/provisioning run/repo/deployment, checklista gotowości sklepu, panel wyboru sklepów, pierwsze dwie sekcje edytora (`hero`, `product_grid`) wbudowane bezpośrednio w `packages/dashboard` (`routes/_authenticated/$storeId/editor.tsx`) — to wszystko realnie działało wg audytu z 14-16 lipca, ale **nie jest kontynuowane**. Właściciel zdecydował po dziewięciu research-passach nad branżowymi wzorcami multi-tenant, że osobne repo/Vercel per sklep to zły kierunek (kosztowna flota do utrzymania przez mały zespół) — nowy model to jeden współdzielony storefront z layoutem jako danymi, silnik dostarcza osobne repo `edytor-sklepu` jako pakiety (edytor w `packages/dashboard/.../editor.tsx` jest więc też legacy — nie rozwijać, docelowy edytor to `/admin` z `edytor-sklepu`, patrz `storefront-composition-system.md`). Kod (`Spree::ProvisioningRun`, `GithubClient`/`VercelClient`, UI w `/$storeId/new-store`, signup flow) zostaje jako legacy, nierozwijany — pełna nota w `docs/plans/store-factory.md`. Etap 0 (bezpieczny fundament multi-tenant backendu — host resolution, tenant-aware cache, role per sklep) i Etap 1 (typy SDK + `@sklepik/test-contracts`) **pozostają aktualne i użyteczne** niezależnie od tej zmiany — to fundament wielosklepowości, nie specyfika repo-per-sklep.
- **Pierwszy krok nowego modelu (2026-07-17): `Spree::StorefrontPage` rozszerzony o pełny słownik 7 sekcji z `edytor-sklepu`'s `component-library`** (wcześniej tylko `hero`/`product_grid`) — patrz `docs/engine-decisions.md` 2026-07-17. **Nieprzetestowane wykonaniem** (brak Ruby/Bundlera/Dockera w tamtej sesji) — uruchomić `bundle exec rspec spec/models/spree/storefront_page_spec.rb` przed poleganiem na tym. Konsument po stronie `edytor-sklepu`: nowy `SklepikPageRepository` (12 testów, zielone, nigdy nie uruchomiony przeciw prawdziwemu backendowi). `sklepikFront` jeszcze nie konsumuje żadnego z tego.

## Znane problemy (aktualne)

Poniższa lista zachowuje szczegóły wcześniejszego stanu i napraw, ale nie jest już kolejnością wykonania. Aktywną kolejność po audytach wyznacza program P0/P1 na początku [`roadmap.md`](roadmap.md); pozycje oznaczone wcześniej jako „zamknięte” pozostają zamknięte tylko w swoim pierwotnym zakresie i nie unieważniają nowych findings przekrojowych.

1. **Dashboard nie pokazuje błędów API (P1, zamknięte, F5 domknięte 2026-07-08):** 500 z backendu przy **ładowaniu list** wcześniej dawał wieczne skeletony zamiast komunikatu — `resource-table.tsx` teraz renderuje `ErrorState` z retry przy `isError`. Osobna, powiązana klasa naprawiona 2026-07-07 (audyt F12, domknięty): **ciche błędy przy mutacjach** — `useOrderMutation` bez `onError` (payment capture/void, fulfillment, zwroty, adresy — sprzedawca mógł myśleć że capture przeszedł, skoro przycisk po prostu przestawał się kręcić), zła invalidacja cache przy edycji adresu, `.catch(() => undefined)` łykający błędy przy usuwaniu klienta, brak obsługi błędu w bulk-add mediów wariantu. Audyt F12 potwierdził trzy martwe/niedostępne z panelu backendowe funkcje (`PriceListsController#prices` bez trasy; `fulfillments#resume`/`#split` bez UI; **kanały dystrybucji nie mają w panelu żadnego sposobu przypisania produktów** — `Channels#add_products`/`#remove_products` istnieją w API i SDK, zero UI). Kontynuacja F13 (2026-07-08) objęła wszystkie pięć promptów i poza jedną małą poprawką store credit nie znalazła nowych cichych mutacji. Katalog zostawił jako decyzje UI/produktu ukryte pola produktowe (`available_on`, `promotionable`, `digital`, `meta_keywords`), brak kosztu własnego wariantu w formularzu i techniczne `metadata` opcji bez edytora (korekta audytu: top-level `/api/v3/admin/variants` jest używany przez kreator transferów). Obszar money-critical zostawił większą lukę: brak Admin API v3/panelu dla shipping methods, shipping categories, zones i tax rates. Obszar security zostawił ryzyko lockoutu: brak backendowego guardu przed usunięciem siebie albo ostatniego administratora sklepu w staff management — **domknięte 2026-07-08 (F14):** `AdminUsersController` odrzuca teraz `destroy`/`update` (odebranie roli `admin`), które zostawiłyby store bez żadnego admina. Audyt pieniędzy klienta potwierdził działające gift cards/customer store credits, ale wskazał brak pełnego admina refunds/returns/reimbursements, wishlist i digital downloads; audyt konfiguracji potwierdził webhooks/custom fields/markets/exports/allowed origins, ale zostawił brak rotacji sekretu webhooka, Admin API/UI dla data feeds i per-wierszowych błędów tłumaczeń. Szczegóły i rekomendacje w `roadmap.md` F12/F13 oraz `audit-playbook.md`.
2. **Cache storefrontu — inwalidacja produktów działa przez webhook, w tym edycja samej ceny (P1, zamknięte, F4):** `Spree::Product` publikuje `product.created`/`updated`/`deleted`/`activated`/`archived`/`out_of_stock`/`back_in_stock` (nie wymagało zmian w backendzie), `sklepikFront`'s `/api/webhooks/spree` obsługuje wszystkie siedem jednym handlerem i busuje cache w sekundach. **Skonfigurowane i aktywne na produkcji** — w adminie (Ustawienia → Webhooks) endpoint na `{storefront}/api/webhooks/spree` ma te siedem eventów w subskrypcji. **Zweryfikowane 2026-07-11:** edycja samej ceny (`Spree::Price`, bez zmiany innego pola produktu) niezawodnie odpala `product.updated` przez łańcuch `touch: true` (Price → Variant → Product) — `Spree::Product` ma `after_touch` callback publikujący event (`spree/core/app/models/spree/product.rb:143`), potwierdzone testem. Ręczny wentyl nadal istnieje: `POST /api/revalidate` (sklepikFront) z `Authorization: Bearer $REVALIDATE_SECRET`. **Uwaga diagnostyczna:** pusty katalog na storefroncie bywa mylony z tym problemem, ale częściej to zupełnie inna przyczyna — patrz punkt 8.
3. **Idempotencja webhooków e-mail (P1, F6 zamknięte kodowo 2026-07-11):** `handlers.ts` używa teraz `src/lib/webhooks/idempotency.ts` z Upstash Redis (TTL 7 dni) zamiast `Set` w pamięci. Aktywne dopiero po ustawieniu `UPSTASH_REDIS_REST_URL`/`UPSTASH_REDIS_REST_TOKEN` (albo `KV_REST_API_URL`/`KV_REST_API_TOKEN`) na Vercelu — bez tego kod łagodnie wraca do starego zachowania in-memory (restart instancji = możliwy duplikat e-maila).
4. **Worker Sidekiq działa (P2, F7 zamknięte 2026-07-09), pre-generowanie wariantów wciąż nie zaimplementowane (F20):** worker działa jako kontener `sidekiq` w Docker Compose na Oracle od migracji hostingu (F8) — problem "jeden proces web dźwiga wszystko" jest rozwiązany. Nadal nic nie enqueue'uje joba pre-generowania wariantów zdjęć zaraz po uploadzie, więc warianty Active Storage nadal generują się leniwie na pierwsze żądanie — zmierzone 12.5s na zimnym cache dla wariantu `xlarge` (2000×2000), 1.3s scache'owane; potrafiło to przekroczyć timeout Vercel Image Optimization i zostawić brak zdjęcia na stronie produktu (mitygacja frontowa: `sklepikFront` woli teraz `large` 720×720 jako główne zdjęcie galerii, `sklepikFront/docs/technical-debt.md` 2026-07-07). Zaimplementowanie joba jest w zakresie F20.
5. **Render free/starter — zamknięte migracją na Oracle (P0, F8 zamknięte 2026-07-09):** problem był realny — cold start ~18 s po bezczynności, OOM (>512 MB) zaobserwowany **dwukrotnie** przy ciężkim ruchu API (drugi raz: 2026-07-07 15:03, ~14 min po zielonym deployu — instancja padła, Render sam podniósł nową, żadnych danych nie stracono). Starter ($7/mo) miał ten sam limit 512 MB co free — nie rozwiązywał OOM. **Rozwiązane migracją na Oracle Cloud VPS** (`VM.Standard.E4.Flex`, 1 OCPU/8 GB RAM) — backend działa tam od 2026-07-09, Render wycofany z produkcji. Szczegóły: `docs/deployment-oracle.md`.
6. **Vercel `sklepik_back` quirk (P2):** webhook potrafi nie łapać pushy; pomaga ręczny Redeploy w UI.
7. **Store name bez tłumaczenia `pl` (P1, przyczyna crasha panelu po loginie — kod naprawiony, dane nie):** `Spree::Store#name` jest tłumaczone przez Mobility, ale wpisane tylko pod `en` ("Shop"). Panel pyta API bez jawnego `locale`, więc dostaje `name: null` pod domyślnym `pl` — `StoreSwitcher` się na tym wywalał (`store.name.split(...)` bez zabezpieczenia), zabierając cały panel do `RouteErrorBoundary`. Kod naprawiony (`getInitials`, PR #15, zmergowany), ale **dane wciąż nie są uzupełnione** — pasek boczny pokazuje fallback (ID sklepu) zamiast prawdziwej nazwy, dopóki ktoś nie ustawi `name` sklepu pod `pl` (przez Admin API: `PATCH /api/v3/admin/store?locale=pl`, analogicznie jak zrobiono to dla 5 z 6 produktów kakao).
8. **`SPREE_API_URL`/`SPREE_PUBLISHABLE_KEY` brakowały na Vercel dla `sklepik_front` (naprawione, ustawione ręcznie):** storefront nigdy się nie łączył z backendem — `isSpreeConfigured()` po cichu zwracał puste odpowiedzi (żaden błąd w logach), co wielokrotnie mylono z problemem cache'a (punkt 2) lub z niepowiązanymi zmianami rynku/waluty/publikacji. **Wniosek na przyszłość:** przy "sklep pokazuje pustkę bez błędu" najpierw sprawdzić te dwie zmienne na Vercelu, zanim zacznie się podejrzewać cache czy dane.

    - **Lekcje na przyszłość:** (a) `docker compose restart` **nie** przeładowuje `.env`/zmiennych z `environment:` — do zmiany env zawsze `docker compose up -d <service>` (recreate). (b) **Żadna usługa nie ma trwałego wolumenu dla Active Storage** — jeśli storage kiedykolwiek spadnie na `:local`, kolejny recreate `web` znów skasuje wgrane w międzyczasie pliki.
