# Stan projektu

**Żywy dokument.** Każdy agent po zakończeniu zadania aktualizuje ten plik tak, żeby odzwierciedlał rzeczywisty, bieżący stan systemu — nie dopisuje kolejnych wpisów dziennika, tylko poprawia treść. Historia jest w gicie.

Ostatnia aktualizacja: 2026-07-11 (F4 zweryfikowane, dokumentacja uspójniona z rzeczywistym stanem infrastruktury). Backend Rails/API + Postgres + Redis + Sidekiq + Nginx/SSL działają w Docker Compose na Oracle VPS (`VM.Standard.E4.Flex`, 1 OCPU, 8 GB RAM, Ubuntu 22.04, Paris) — produkcja od 2026-07-09 (F8), Render wycofany. Vercel (admin panel i storefront) zaktualizowany — API URL zmieniony z Rendera na Oracle (141.253.103.172). Razem z audytem F13 system ma werdykt: Not production-ready (główne blokery: checkout payment, braki prawno-operacyjne — werdykt sprzed migracji na Oracle, infrastrukturalne OOM/cold-start bloker już nieaktualny). Domknięte: F4 (webhook przy edycji ceny, 2026-07-11), F5 (jawne błędy w ResourceTable, 2026-07-08), F7 (worker Sidekiq, 2026-07-09), F8 (migracja na Oracle, 2026-07-09), F14 (lockout guard, 2026-07-08), F15 (migration idempotency, 2026-07-10), F16 (rate limiting, 2026-07-10), F17 (webhook secret rotation, 2026-07-10), F18 (per-row translation errors, 2026-07-10), F21 (shipping/tax/zones UI, 2026-07-10). Backend testy (RSpec core/api/emails) tymczasowo wyłączone na pull requestach od 2026-07-11, patrz punkt 13 niżej. **Naprawione 2026-07-10:** PR #25 — Rack::Attack initializer brakował `require 'rack/attack'` (powodował NameError na boot, co blokował migracje na wszystkich 627 testach); role_user_spec.rb na MySQL używała hardcoded ID 99 powodując duplicate key error na parallel test runs (zmienione na timestamp-based unique ID generation).

## Co działa

- **Backend na Oracle Cloud** (141.253.103.172) buduje się i działa: Docker Compose (Rails/Puma + Postgres + Redis + Sidekiq + Nginx), gemy z tego forka (`SPREE_PATH`), migracje silnika zainstalowane i wykonane, baza zseedowana, SSL przez self-signed cert (docelowo Let's Encrypt + domena).
- **Panel admina** (`sklepik-gamma.vercel.app`) działa end-to-end: logowanie JWT przez single-origin proxy, zarządzanie produktami. Wpisy cen kanoniczny format `"1234.56"` niezależnie od locale. Upload logo sklepu (Ustawienia → Sklep) działa (F10).
- **Storefront** (`sklepikkk.vercel.app`) działa: rebranding "Kakałowy Sklepik", polski domyślny locale bez prefiksu URL, katalog i strony produktów renderują się z danych Store API. Logo sklepu (gdy wgrane) renderuje się w nagłówku i JSON-LD z publicznego Store API zamiast statycznego env (F10). Wymaga `SPREE_API_URL` + `SPREE_PUBLISHABLE_KEY` ustawionych jako zmienne środowiskowe Vercel produkcyjnie — bez nich `isSpreeConfigured()` po cichu zwraca puste odpowiedzi (żadnego błędu, katalog wygląda po prostu na pusty; to właśnie się stało i kosztowało długie dochodzenie, zanim się okazało że przyczyną nie był cache).
- **Dane:** jeden rynek Polska/PLN/pl (7 demo-rynków usuniętych), 6 produktów kakao zseedowanych przez Admin API, ceny w PLN, media na R2.
- **Walidacja produktu:** endpoint `/api/v3/admin/products/:id/readiness` sprawdza status, publikację, ceny, stock, tłumaczenia; strona edycji produktu w dashboardzie pobiera go (`useProductReadiness`) i pokazuje banner ostrzegawczy z listą niespełnionych warunków, gdy `ready: false` — żadna z tych kontroli nie blokuje zapisu, to czysto informacyjne ostrzeżenie.
- Testy storefrontu zielone (build + 89 testów vitest na moment rebrandingu).

## Znane problemy (aktualne)

Uporządkowane wg wagi — szczegóły i plan naprawy w [`roadmap.md`](roadmap.md):

1. **Dashboard nie pokazuje błędów API (P1, zamknięte, F5 domknięte 2026-07-08):** 500 z backendu przy **ładowaniu list** wcześniej dawał wieczne skeletony zamiast komunikatu — `resource-table.tsx` teraz renderuje `ErrorState` z retry przy `isError`. Osobna, powiązana klasa naprawiona 2026-07-07 (audyt F12, domknięty): **ciche błędy przy mutacjach** — `useOrderMutation` bez `onError` (payment capture/void, fulfillment, zwroty, adresy — sprzedawca mógł myśleć że capture przeszedł, skoro przycisk po prostu przestawał się kręcić), zła invalidacja cache przy edycji adresu, `.catch(() => undefined)` łykający błędy przy usuwaniu klienta, brak obsługi błędu w bulk-add mediów wariantu. Audyt F12 potwierdził trzy martwe/niedostępne z panelu backendowe funkcje (`PriceListsController#prices` bez trasy; `fulfillments#resume`/`#split` bez UI; **kanały dystrybucji nie mają w panelu żadnego sposobu przypisania produktów** — `Channels#add_products`/`#remove_products` istnieją w API i SDK, zero UI). Kontynuacja F13 (2026-07-08) objęła wszystkie pięć promptów i poza jedną małą poprawką store credit nie znalazła nowych cichych mutacji. Katalog zostawił jako decyzje UI/produktu ukryte pola produktowe (`available_on`, `promotionable`, `digital`, `meta_keywords`), brak kosztu własnego wariantu w formularzu i techniczne `metadata` opcji bez edytora (korekta audytu: top-level `/api/v3/admin/variants` jest używany przez kreator transferów). Obszar money-critical zostawił większą lukę: brak Admin API v3/panelu dla shipping methods, shipping categories, zones i tax rates. Obszar security zostawił ryzyko lockoutu: brak backendowego guardu przed usunięciem siebie albo ostatniego administratora sklepu w staff management — **domknięte 2026-07-08 (F14):** `AdminUsersController` odrzuca teraz `destroy`/`update` (odebranie roli `admin`), które zostawiłyby store bez żadnego admina. Audyt pieniędzy klienta potwierdził działające gift cards/customer store credits, ale wskazał brak pełnego admina refunds/returns/reimbursements, wishlist i digital downloads; audyt konfiguracji potwierdził webhooks/custom fields/markets/exports/allowed origins, ale zostawił brak rotacji sekretu webhooka, Admin API/UI dla data feeds i per-wierszowych błędów tłumaczeń. Szczegóły i rekomendacje w `roadmap.md` F12/F13 oraz `audit-playbook.md`.
2. **Cache storefrontu — inwalidacja produktów działa przez webhook, w tym edycja samej ceny (P1, zamknięte, F4):** `Spree::Product` publikuje `product.created`/`updated`/`deleted`/`activated`/`archived`/`out_of_stock`/`back_in_stock` (nie wymagało zmian w backendzie), `sklepikFront`'s `/api/webhooks/spree` obsługuje wszystkie siedem jednym handlerem i busuje cache w sekundach. **Skonfigurowane i aktywne na produkcji** — w adminie (Ustawienia → Webhooks) endpoint na `{storefront}/api/webhooks/spree` ma te siedem eventów w subskrypcji. **Zweryfikowane 2026-07-11:** edycja samej ceny (`Spree::Price`, bez zmiany innego pola produktu) niezawodnie odpala `product.updated` przez łańcuch `touch: true` (Price → Variant → Product) — `Spree::Product` ma `after_touch` callback publikujący event (`spree/core/app/models/spree/product.rb:143`), potwierdzone testem. Ręczny wentyl nadal istnieje: `POST /api/revalidate` (sklepikFront) z `Authorization: Bearer $REVALIDATE_SECRET`. **Uwaga diagnostyczna:** pusty katalog na storefroncie bywa mylony z tym problemem, ale częściej to zupełnie inna przyczyna — patrz punkt 8.
3. **Idempotencja webhooków e-mail (P1, F6 zamknięte kodowo 2026-07-11):** `handlers.ts` używa teraz `src/lib/webhooks/idempotency.ts` z Upstash Redis (TTL 7 dni) zamiast `Set` w pamięci. Aktywne dopiero po ustawieniu `UPSTASH_REDIS_REST_URL`/`UPSTASH_REDIS_REST_TOKEN` (albo `KV_REST_API_URL`/`KV_REST_API_TOKEN`) na Vercelu — bez tego kod łagodnie wraca do starego zachowania in-memory (restart instancji = możliwy duplikat e-maila).
4. **Worker Sidekiq działa (P2, F7 zamknięte 2026-07-09), pre-generowanie wariantów wciąż nie zaimplementowane (F20):** worker działa jako kontener `sidekiq` w Docker Compose na Oracle od migracji hostingu (F8) — problem "jeden proces web dźwiga wszystko" jest rozwiązany. Nadal nic nie enqueue'uje joba pre-generowania wariantów zdjęć zaraz po uploadzie, więc warianty Active Storage nadal generują się leniwie na pierwsze żądanie — zmierzone 12.5s na zimnym cache dla wariantu `xlarge` (2000×2000), 1.3s scache'owane; potrafiło to przekroczyć timeout Vercel Image Optimization i zostawić brak zdjęcia na stronie produktu (mitygacja frontowa: `sklepikFront` woli teraz `large` 720×720 jako główne zdjęcie galerii, `sklepikFront/docs/technical-debt.md` 2026-07-07). Zaimplementowanie joba jest w zakresie F20.
5. **Render free/starter — zamknięte migracją na Oracle (P0, F8 zamknięte 2026-07-09):** problem był realny — cold start ~18 s po bezczynności, OOM (>512 MB) zaobserwowany **dwukrotnie** przy ciężkim ruchu API (drugi raz: 2026-07-07 15:03, ~14 min po zielonym deployu — instancja padła, Render sam podniósł nową, żadnych danych nie stracono). Starter ($7/mo) miał ten sam limit 512 MB co free — nie rozwiązywał OOM. **Rozwiązane migracją na Oracle Cloud VPS** (`VM.Standard.E4.Flex`, 1 OCPU/8 GB RAM) — backend działa tam od 2026-07-09, Render wycofany z produkcji. Szczegóły: `docs/deployment-oracle.md`.
6. **Vercel `sklepik_back` quirk (P2):** webhook potrafi nie łapać pushy; pomaga ręczny Redeploy w UI.
7. **Store name bez tłumaczenia `pl` (P1, przyczyna crasha panelu po loginie — kod naprawiony, dane nie):** `Spree::Store#name` jest tłumaczone przez Mobility, ale wpisane tylko pod `en` ("Shop"). Panel pyta API bez jawnego `locale`, więc dostaje `name: null` pod domyślnym `pl` — `StoreSwitcher` się na tym wywalał (`store.name.split(...)` bez zabezpieczenia), zabierając cały panel do `RouteErrorBoundary`. Kod naprawiony (`getInitials`, PR #15, zmergowany), ale **dane wciąż nie są uzupełnione** — pasek boczny pokazuje fallback (ID sklepu) zamiast prawdziwej nazwy, dopóki ktoś nie ustawi `name` sklepu pod `pl` (przez Admin API: `PATCH /api/v3/admin/store?locale=pl`, analogicznie jak zrobiono to dla 5 z 6 produktów kakao).
8. **`SPREE_API_URL`/`SPREE_PUBLISHABLE_KEY` brakowały na Vercel dla `sklepik_front` (naprawione, ustawione ręcznie):** storefront nigdy się nie łączył z backendem — `isSpreeConfigured()` po cichu zwracał puste odpowiedzi (żaden błąd w logach), co wielokrotnie mylono z problemem cache'a (punkt 2) lub z niepowiązanymi zmianami rynku/waluty/publikacji. **Wniosek na przyszłość:** przy "sklep pokazuje pustkę bez błędu" najpierw sprawdzić te dwie zmienne na Vercelu, zanim zacznie się podejrzewać cache czy dane.
9. **Produkty bez `product_publications` = niewidoczne (P1, ryzyko powtórki):** produkt utworzony przez Admin API bez jawnego powiązania z kanałem (`product_publications`) jest niewidoczny na Store API mimo `status: active`, `available_on` w przeszłości i towaru na stanie. `Spree::Products::ReadinessCheck` (F3) to teraz wykrywa, ale nic go automatycznie nie wywołuje przy tworzeniu produktu przez API — łatwo to przeoczyć przy każdym kolejnym ręcznym seedowaniu danych.
10. **Przełącznik waluty/kraju w dashboardzie panelu pusty (P2, częściowo sprawdzone kodowo):** po usunięciu 6 z 7 demo-rynków (patrz "Co działa") lista krajów/walut do wyboru w UI panelu była zgłaszana jako pusta lub niekompletna (widać PLN/USD, ale nie da się wybrać). Audyt F13 potwierdził kodowo, że formularz `settings/markets` używa pełnego selecta walut i comboboxa krajów, ale nie reprodukował działającego panelu w przeglądarce, więc zgłoszenia przełącznika kraju/waluty nie zamykamy bez manualnej weryfikacji runtime.
11. **Przełącznik kraju/waluty w storefroncie — częściowo naprawiony (P1, F11, kroki 0+1 zamknięte 2026-07-07):** zepsuty `CountrySwitcher.tsx` (404 na wyborze kraju, wizualny duplikat) usunięty, zastąpiony `LanguageSwitcher.tsx` — czysto językowym przełącznikiem niezależnym od waluty. Prawdziwy przełącznik rynku/waluty (`MarketSwitcher`) wciąż nieistnieje — czeka na realny drugi `Market` w adminie. Plan: [`docs/plans/market-language-switcher.md`](plans/market-language-switcher.md).
12. **Legacy Rails admin (`spree/admin`) wyłączony, ale jego specy wisiały w CI (naprawione):** trasy legacy panelu są wyłączone na rzecz React dashboardu, więc jego specy kontrolerów/feature/request nie mają już helperów tras i failowały masowo (~1000 na `main`). Suite `admin` została usunięta z macierzy `tests.yml` (obie bazy) do czasu jej usunięcia; `core`/`api`/`emails` bez zmian. Przy okazji naprawione dwa realne braki w `api`: brakujący przykład SDK `packages/sdk/examples/store/get.ts` (blokował ładowanie `store_spec.rb`) oraz specy cen pod locale `:de`, które tworzyły produkt z tłumaczonym `name` w niewłaściwej kolejności.
13a. **Postgres/Redis na Oracle VPS bywają zatrzymywane przez nieznane, równoległe źródło (P1, obserwowane 2026-07-11):** kontenery `postgres`/`redis` kilkukrotnie zostały czysto zatrzymane (`docker stop`, exit 0) kilka sekund po połączeniu SSH z IP spoza znanych adresów właściciela, używającym tego samego klucza prywatnego. Przyczyna nieustalona (możliwe: inna, równoległa sesja agenta/narzędzia z dostępem do tego samego klucza — właściciel potwierdził, że nie wie co to jest i nie ma nad tym kontroli). **Mitygacja:** zainstalowano watchdog (`/home/ubuntu/db-watchdog.sh` + cron co minutę, log w `/home/ubuntu/db-watchdog.log`), który automatycznie odpala `docker compose up -d postgres redis`, jeśli oba albo jeden z nich nie działają — ogranicza przestój do ok. 1 minuty zamiast trwałego down. To nie usuwa przyczyny, tylko skraca skutki. Objaw dla agentów: jeśli backend nagle zwraca 500 na czymkolwiek wymagającym bazy, **najpierw** `docker compose ps` na serwerze, nie zakładaj od razu błędu w kodzie/konfiguracji.

13. **Backend tests (RSpec core/api/emails) TYMCZASOWO wyłączone na pull_request'ach (P0, 2026-07-11):** testy brały 30-40 minut na shard (MySQL/PostgreSQL), co blokowało development velocity. Decyzja: wyłączyć testy na PR'ach (`.github/workflows/tests.yml` — warunek `if: github.event_name == 'push'`), uruchamiać tylko przy push'u do `main` przed release'em. **TODO przed production launch:** ponownie włączyć i upewnić się że suite przechodzi na zielono.

14. **Operacyjne twardnienie produkcji — częściowo zrobione (zidentyfikowane 2026-07-11):**
    - **Backup Postgresa — ZROBIONE 2026-07-11:** `/home/ubuntu/backup-postgres.sh` (`pg_dumpall` → gzip → lokalnie `/home/ubuntu/backups/` **i** upload do R2 przez `aws s3 cp` z istniejącymi `CLOUDFLARE_*` credentials z `.env`), cron codziennie o 3:00 UTC, log w `/home/ubuntu/backup-postgres.log`, lokalne kopie starsze niż 7 dni czyszczone automatycznie. Przetestowany ręcznym uruchomieniem — dump 61K, upload na R2 potwierdzony. **Nadal nie przetestowano pełnego restore** — zrobić to przed poleganiem na tym w razie realnej awarii.
    - **Domyślne hasło admina — ZROTOWANE 2026-07-11:** `admin@admin.pl` ma teraz losowe 24-znakowe hasło (przekazane właścicielowi w czacie, nigdzie niezapisane na dysku ani w repo). Zweryfikowane działającym loginem przez `sklepik-gamma.vercel.app`.
    - **Sekrety (`RAILS_MASTER_KEY`, `SECRET_KEY_BASE`, hasła DB, tokeny R2) w plikach `.env`/`docker-compose.yml` na serwerze bez menedżera sekretów** — czytelne w plaintext dla każdego z dostępem do VPS. Nie jest to pilne przy jednym zaufanym operatorze, ale przed dodaniem kolejnych osób z SSH-dostępem trzeba to przenieść do Docker secrets albo Vault-podobnego rozwiązania. **Nienaprawione — wymaga decyzji właściciela**, nie jest to coś do zrobienia bez pytania.
    - **Monitoring/alerty — częściowo zrobione 2026-07-11:** `/home/ubuntu/healthcheck.sh` (cron co 5 min) sprawdza stan wszystkich 5 kontenerów, `GET /up` i zapełnienie dysku, loguje do `/home/ubuntu/healthcheck.log`. **Wysyłka e-mail NIE działa** — `SMTP_HOST`/`SMTP_USER`/`SMTP_PASSWORD` w `.env` to wciąż placeholdery (`smtp.example.com`), nie prawdziwe dane; potwierdzone testem (`curl: Could not resolve host`). Właściciel świadomie odłożył to na później (2026-07-11) zamiast podać dane SMTP albo skonfigurować ntfy.sh. Dopóki `.env` ma placeholder SMTP, healthcheck jest tylko logiem do ręcznego sprawdzenia (`cat /home/ubuntu/healthcheck.log`), nie realnym alertem. **Do zrobienia:** podać prawdziwe dane SMTP (albo inny kanał) i zweryfikować wysyłkę.
    - **Klucz SSH do VPS jest współdzielony między operatorem a automatyzacją** (CI/CD lub monitoring logujący się z tego samego klucza dziesiątki/setki razy dziennie — zaobserwowane IP z zakresu `31.60.*.*` logujące się nawet 235 razy). Jeden skompromitowany klient tego klucza = pełny dostęp roota do produkcji. Zalecane: osobny klucz dla automatyzacji z ograniczonymi uprawnieniami (`authorized_keys` z `command=`/`no-port-forwarding`), rotacja kluczy co jakiś czas. **Świadomie nienaprawione przez agenta** — to zmiana kontroli dostępu, robi ją tylko właściciel.

## Czego jeszcze nie ma (przed startem sprzedaży)

- System-wide production readiness audit (2026-07-08) ma werdykt `Not production-ready`; szczegóły i priorytety napraw są w `docs/audits/2026-07-08-system-wide-production-readiness-audit.md`.

- Strony prawne: regulamin, polityka prywatności, prawo odstąpienia (wymagane w PL) — świadomie odłożone.
- Własna domena (wszystko na `*.vercel.app` dla storefrontu/panelu, Oracle Cloud IP dla backendu).
